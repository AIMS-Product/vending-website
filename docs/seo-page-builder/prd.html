01

Executive Summary

Vendingpreneurs needs a repeatable structure for creating high-quality SEO resource pages over time. This is not a one-off content batch; it is a controlled visual page-building system that lets admins compose pages from reusable, validated blocks.

Core Thesis

Build a structured WYSIWYG page builder that gives admins creative speed while preserving brand quality, SEO safety, conversion tracking, and publishing control.

Critical Constraint

AI can assist with structure, copy, metadata, schema, and internal links, but it must only use approved internal source material and must never publish directly.

02

The Problem

# Problem Today
P1 No scalable page creation surface SEO pages would require code edits or one-off implementation work, which slows iteration and makes page volume hard to scale.
P2 Freeform AI/page generation is risky Without source controls, validation, and human approval, generated SEO pages could drift off-brand or make unsupported claims.
P3 SEO and conversion requirements need enforcement Metadata, schema, alt text, redirects, sitemap behavior, CTA presence, and lead attribution must be reliable before pages go live.
03

Goals & Non-Goals

Goal 01

Visual creation. Admins can create SEO pages with rows, columns, inline editing, drag/drop ordering, and brand presets.

Goal 02

Structured content. Pages are stored as validated block data, not arbitrary HTML.

Goal 03

SEO-safe publishing. Every public page has metadata, sitemap behavior, schema controls, and redirect handling.

Goal 04

Conversion tracking. CTA and lead form blocks automatically attribute submissions to page, keyword, source block, UTM, and referrer data.

Non-Goals (v1)

  • Do not build a freeform Webflow-style design canvas.
  • Do not allow arbitrary colors, fonts, margins, or custom HTML blocks in the first version.
  • Do not allow AI to publish pages directly.
  • Do not create the first SEO content batch as part of the foundation slice.
04

Solution

The product should extend the existing Supabase-backed admin surface into a visual page builder. Draft pages are edited as a row/column/block tree, autosaved, previewed through secret links, validated, and then manually published to /resources/[slug].

Builder flow The expected admin workflow
1
Admin creates a new SEO resource page and sets title, slug, target keyword, and SEO metadata.
2
Admin adds rows, columns, and approved blocks, then edits content inline and reorders with drag/drop.
3
Optional AI proposal uses selected approved source material to suggest blocks, schema, metadata, and internal links.
4
Accepted AI suggestions are inserted into the draft; rejected suggestions are discarded without changing the page.
5
Autosave persists draft state. Publish validation must pass before the public snapshot updates.
Recommended Build Order

Ship an end-to-end thin slice first: text, image, CTA blocks, draft autosave, publish snapshot, public render, sitemap inclusion, and hard publish validation.

07

Success Criteria

1
End-to-end creation works. An admin can create, edit, autosave, preview, publish, and view a live resource page.
2
Structured blocks render safely. Public pages and editor previews use the same block registry and pass validation.
3
Publishing is guarded. Missing metadata, invalid blocks, missing alt text, no CTA/form, broken internal links, or schema errors block publish.
4
Revisions are recoverable. Admins can preview an old revision and restore it into draft without changing the published page until republish.
5
SEO routing behaves correctly. Published pages appear in sitemap, archived pages do not, and published slug changes create 301 redirects.

09

Phasing

Slice Scope Depends on
1 critical End-to-end builder foundation: migrations, page list/create/edit, rows/columns, rich text, image, CTA, autosave, publish, public render, sitemap, publish validation. Existing admin/auth model and Supabase project.
2 Revision UI and preview links: revision list, revision preview, rollback to draft, secret preview tokens, revocation/expiry. Slice 1 content snapshots.
3 Media library: upload, tags/search, alt text, source/rights metadata, image block asset references. Slice 1 image block path.
4 Full core blocks: hero, video, FAQ, card grid, proof, lead form, FAQ schema and breadcrumbs. Stable block registry.
5 Reusable content libraries: proof items, CTA presets, source documents, excerpts, approved claims, draft refresh flow. Media library and revision snapshots.
6 AI proposal workflow: proposed block tree, suggested metadata/schema, source references, warnings, accept-selected insertion. Source libraries and block validation.
7 later SEO hardening: internal link suggestions, duplicate metadata checks, SERP preview, redirect conflict UI. Published page inventory.
10

Out of Scope

Explicit exclusions

  • Custom freeform design controls beyond approved brand presets.
  • Root-level dynamic pages that could collide with existing routes.
  • Automated publishing or auto-updating live pages from library changes.
  • Live web research for AI-generated page claims in v1.
  • A/B testing infrastructure in the foundation slice.

Verification

End-to-end post-implementation check:

01
Create a draft page with valid slug, SEO title, meta description, target keyword, one rich text block, one image block, and one CTA block.
02
Confirm autosave persists draft state and public visitors cannot access the draft route.
03
Publish the page and confirm it renders at /resources/[slug] and appears in sitemap when eligible.
04
Change the slug after publish and confirm the old URL returns a 301 redirect to the new URL.
05
Restore an older revision into draft and confirm the published page does not change until republished.
06
Attempt to publish with missing metadata, missing image alt text, or no CTA/form and confirm publish is blocked.
