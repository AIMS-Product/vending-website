## Product decisions implemented

- Super_admin-only mutations: actions call `requireSuperAdmin()` (same gating helper as Settings > Users); page renders read-only banner + disabled form for plain admins. RLS uses `is_app_admin()` like all sibling admin tables; the curated-mutation gate is the server-action layer, identical to the Users page model.
- Prefix shape `^/[a-z0-9]+(-[a-z0-9]+)*$`, max 40 — enforced in Zod (service) AND as a DB CHECK constraint.
- Input normalization: trim, lowercase, auto-prepend `/`.
- `is_default` rows refuse deletion in the service (`Default route prefixes cannot be deleted.`) and have no delete button in the UI.
- Duplicate prefix (23505) surfaces as "That route prefix already exists."

## Residual risk

1. The action-layer state type is `{ status: "idle" | "saved" | "error"; message }`, mirroring the existing `UserSettingsActionState` idiom rather than the literal `{ success, error? }` shape named in the node — chose consistency with the sibling settings page it was told to mirror.
2. `create policy` is not idempotent (no `drop policy if exists`) — same as the June sibling migrations; only matters if the migration is partially applied then re-run.
3. UI delete has no confirm dialog (deleting a custom prefix is low-stakes and recoverable by re-adding); Users page uses one for access removal — can be added if review wants parity.
4. Defaults sorted by `created_at asc` — seed order is preserved in practice but ties on identical timestamps could reorder; cosmetic only.
