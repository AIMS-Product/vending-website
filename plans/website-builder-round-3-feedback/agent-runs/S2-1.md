## Behavior preservation

- **Whole-paragraph fallback:** empty "Link text" + URL produces `spans: [{ text, href }]` exactly as before (test: "falls back to the whole-paragraph link when link text is empty"). Clearing the URL returns a plain `{ type: "paragraph", text }` node (test covered).
- **Unsafe href:** `isSafeLinkHrefInput` rule unchanged (empty | root-relative non-`//` | http(s)); `buildParagraphLinkNode` returns `{ ok: false, error: "unsafe_href" }` for `javascript:`, `//host`, `ftp:`, bare-relative — tested. The href input keeps its existing silent-ignore-while-typing behavior.
- **Edit re-derivation (documented decision):** editing the link controls re-derives spans from the paragraph's plain text (the existing model). Editing the paragraph _text_ still collapses to a whole-paragraph link via the pre-existing `updateNodeText` (unchanged) — author re-applies the link text after a body edit. Local "Link text" input state initializes from an existing multi-span link (`paragraphLinkText`), shows empty for legacy whole-paragraph links.

## Residual risk

- `RichTextBodyEditor.tsx` is now 395 lines (was 325) — over the repo 300-line guideline, but the write scope confined the UI to this file. A follow-up could move `ParagraphLinkControls` to its own file.
- Error display and typing interactions are unit-covered at the helper level only (node test env, no jsdom); the inline error and live split need the orchestrator's browser gate for visual confirmation.
- If the author edits paragraph text after creating a substring link, the link widens back to the whole paragraph (pre-existing `updateNodeText` model, explicitly preserved per node spec). Worth a tester note.
