# S6b-2 — Fix: custom-prefix redirect rows must be real HTTP 3xx (addendum to S6b-1)

**Status:** COMPLETE — root-caused with doc citations, fixed in the proxy, live-verified read-only (HTTP 301 + Location header on the running dev server), all gates pass.

## Root cause (confirmed, not hypothesis)

The streaming hypothesis is **confirmed** with live evidence and doc citations:

1. **Live body inspection** — `curl http://localhost:3000/services/r3-services-routing-proof` returned HTTP 200 (no `Location` header) and the streamed HTML contained:
   - `<meta id="__next-page-redirect" http-equiv="refresh" content="0;url=/services/r3-services-proof-renamed"/>` (×2)
   - 10 occurrences of `NEXT_REDIRECT` digests in the RSC payload
   - the not-found shell UI in the same stream

   So `permanentRedirect()` in `[...builderPath]/page.tsx` _did_ fire and _did_ find the row — but the response status was already committed.

2. **Doc citations** (Next 16.2.6 bundled docs):
   - `node_modules/next/dist/docs/01-app/03-api-reference/04-functions/permanentRedirect.md` line 11: "**When used in a streaming context, this will insert a meta tag to emit the redirect on the client side.** When used in a server action, it will serve a 303… Otherwise, it will serve a 308 (Permanent) HTTP redirect response."
   - `redirect.md` line 11: identical caveat for `redirect()` (meta tag in streaming context, 307 otherwise).

3. **Why this route always streams:** the route is fully dynamic (`revalidate = 0`) and sits under loading boundaries (root `src/app/loading.tsx` + the route's own `loading.tsx`), so Next flushes a 200 shell immediately; any redirect thrown afterwards (in `generateMetadata` or the page body) can only be a client-side meta refresh. Browsers follow it; curl/crawlers see a 200 soft page. This is exactly why the five default prefixes have always had their redirect rows served by the **proxy** (pre-routing, real `NextResponse.redirect`) — custom prefixes simply weren't covered because the proxy matcher is a static five-prefix list.

## Fix

Serve custom-prefix redirect rows in `src/proxy.ts`, the same mechanism the five defaults already use:

- **Matcher** (`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md` lines 73–135: path-to-regexp with custom regex in parens; values must be static constants): added `"/:prefix([a-z0-9-]+)/:slug([^/]+)"` — admits every two-segment lowercase-kebab path, the only shape a builder `route_path` can take. `_next/*` (underscore), uppercase paths, and 1- or 3-segment paths don't match.
- **Terminal in-function branch** placed immediately after the existing default-prefix block:
  - `isCustomBuilderPathCandidate(path)`: exactly two segments AND not `/admin/`, `/auth/`, `/news/` (those keep flowing to their existing branches byte-identically).
  - `handleCustomBuilderPath(request, path)`:
    1. `splitAssignableBuilderRoutePath(path)` — shape + reservation gate; reserved segments (`/authors`, `/images`, `/api`, …) and non-kebab paths return `NextResponse.next()` with **zero DB lookups**.
    2. `listRoutePrefixes()` — live configured-list check; unconfigured prefixes return `next()` (one indexed query on a tiny table; same query the page itself already runs).
    3. `getBuilderRedirectBySourcePath(path)` → `NextResponse.redirect(resolveRedirectDestination(...), status_code)` — identical to the default-prefix branch.
    4. Otherwise `next()` — existence/404 handling deliberately stays with the route (per scope: only redirects need real 3xx; the site-wide streamed soft-404 pattern is untouched).
  - The branch is **terminal for every path the new matcher admits** — nothing new can ever fall through to the `updateSession`/admin auth gate (this was the main hazard of broadening the matcher; pinned by test).
- The in-page `applyBuilderRedirects` in `[...builderPath]/page.tsx` is kept unchanged as a browser-level fallback (meta refresh) for any request path that bypasses the proxy.

## Default-prefix redirect serving — unaffected (code-read)

The five-prefix `isBuilderRoutePath(path)` block runs **before** the new branch and is unchanged; `isCustomBuilderPathCandidate` is only reachable for paths that block didn't consume. The new matcher entry overlaps the existing `/resources/:path*` etc. entries, which is harmless (matcher entries are OR'd; the function runs once). Pinned by the existing three legacy-blog proxy tests, the existing 404/redirect tests, and the new "keeps default-prefix redirect rows served exactly as before" test.

## Residual risk

1. Proxy now runs on all two-segment kebab paths (incl. `/api/{x}` and junk URLs). Non-candidates exit before any DB call (regex + Set check, ~1–2ms observed); configured-prefix paths add the same `listRoutePrefixes` query the page already performs. No auth-flow paths changed (admin/auth/news excluded explicitly; uppercase/`_next` never match the matcher shape).
2. Missing/unpublished custom-prefix pages still return a streamed 200 soft-404 (unlike the five defaults' proxy-level 404). Out of scope per instruction; trivial follow-up if wanted — add a `hasPublishedSeoPagePath` check in `handleCustomBuilderPath` mirroring the default branch.
3. The matcher regex was verified live in dev; Vercel production uses the same compiled matcher (build emits it in the middleware manifest), but post-deploy re-verification of one custom-prefix redirect is recommended.
