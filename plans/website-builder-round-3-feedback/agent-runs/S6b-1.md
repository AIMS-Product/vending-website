## Route-precedence evidence (Next 16.2.6 docs + source)

Read per AGENTS.md before writing code:

- `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/dynamic-routes.md` — catch-all conventions, runtime param validation pattern (`notFound()` for invalid params)
- `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/route-groups.md` — groups are URL-transparent; conflicting-path caveat
- `node_modules/next/dist/build/validate-app-paths.js` — build-time validation normalizes param names to `[*]`; only structurally equivalent routes conflict
- `node_modules/next/dist/server/route-matcher-managers/default-route-matcher-manager.js` (line 112) — the **runtime** matcher orders dynamic routes with `getSortedRoutes`
- `node_modules/next/dist/shared/lib/router/utils/sorted-routes.js` (line ~128) — `getSortedRoutes` throws E337 ("You cannot use different slug names for the same dynamic path") for two different param names at the same dynamic tree position

**Documented deviation from the suggested `[prefix]/[slug]` shape:** `src/app/[legacyLeadPath]/page.tsx` already occupies the top-level dynamic slot. A sibling `[prefix]` segment throws E337 in the runtime sorter (proven by a test: `getSortedRoutes(["/[legacyLeadPath]", "/[prefix]/[slug]"])` throws `/different slug names/`). A root **catch-all** `[...builderPath]` occupies the separate `restSlugName` slot, coexists, and is the _lowest-priority_ route — static routes, `/news/[slug]`, `/resources/preview/[token]`, and `[legacyLeadPath]` all win. The page enforces exactly-2-segments + configured-prefix, `notFound()` otherwise (matches previous 404 behavior for unmatched paths).

Build route table confirms coexistence:

```
├ ƒ /[...builderPath]
├ ƒ /[legacyLeadPath]
...
├ ● /news/[slug]
├ ƒ /resources/preview/[token]
```

`route-precedence.test.ts` additionally asserts (with the real sorter) that `/resources/preview/[token]`, `/news/[slug]`, `/authors/[slug]`, `/admin/pages/[id]`, and `/[legacyLeadPath]` all sort above `/[...builderPath]`, and that the catch-all sorts last.

## Residual risk

1. **Custom prefixes bypass the proxy** (its matcher is statically limited to the five defaults + news/admin/auth). Mitigated: the catch-all page itself resolves redirect rows for non-default prefixes (`getBuilderRedirectBySourcePath` → 308/307) and 404s unknown slugs via `notFound()`. The proxy's pre-route 404 (bot-friendly minimal HTML) doesn't apply to custom prefixes — they get the app's standard `not-found.tsx` with a 404 status instead. Functionally equivalent; slightly heavier response.
2. **Unmatched multi-segment URLs now execute the catch-all** (one `listRoutePrefixes` query) before 404ing, instead of 404ing in the router. Same status/page for users; small extra DB read on junk URLs.
3. **Per-request prefix lookups** run in both `generateMetadata` and the page (2 queries/render) — consistent with the existing duplicated `getPublishedSeoPageByPath` pattern in the old routes.
4. Browser-level proof (custom prefix end-to-end with a real published page) not run — forbidden for this node.
