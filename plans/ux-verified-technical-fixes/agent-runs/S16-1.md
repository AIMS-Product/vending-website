## Check A — C132 landmark-unique on `/admin/pages/new`

**Verdict: REPRODUCED** (moderate `landmark-unique` still fires after the one-step rewrite).

Evidence:

- axe JSON: `plans/ux-verified-technical-fixes/agent-runs/s16-axe-pages-new.json`
- screenshot: `plans/ux-verified-technical-fixes/agent-runs/shots/s16-a-pages-new.png`
- axe result: exactly **1** violation, id `landmark-unique`, impact **moderate**, 1 node.
- Offending node (from JSON `failureSummary`):
  `<section aria-labelledby="admin-shell-title" class="min-w-0 p-0">` (target `.min-w-0`).
  > "The landmark must have a unique aria-label, aria-labelledby, or title to make landmarks distinguishable."

Root cause: two named `region` landmarks render on this route and axe cannot
distinguish them as a unique role/label pair. The AdminShell content wrapper is
a `<section aria-labelledby="admin-shell-title">`
(`src/components/admin/AdminShell.tsx`), and the create-gate panel inside it is
another `<section aria-labelledby="new-page-choice-title">`
(`src/components/admin/seo-page-editor/SeoPageEditorShell.tsx:76-78`). Because a
`<section>` with an accessible name is exposed as a `region` landmark, the page
has nested same-role landmarks, which trips `landmark-unique`.

Fix sketch: the gate panel doesn't need to be a landmark `<section>` — drop the
`aria-labelledby` (or change the outer `<section aria-labelledby="...">` to a
plain `<div role="presentation">` / non-landmark wrapper) so only one region
landmark exists, or give the inner panel a `<section aria-label>` that is
genuinely distinct AND ensure the shell wrapper isn't also a region. Cheapest
safe fix: make the inner create-gate container a `<div>` (it still has the
visible `<h2 id="new-page-choice-title">` heading for structure) rather than a
named `<section>`. Single-file change in `SeoPageEditorShell.tsx`; re-run axe to
confirm 0 violations.

## Check B — C145 news-editor "Save draft" visibility

**Verdict: REPRODUCED** ("Save draft" is below the fold and non-visible during editing at both 375px and 768px).

Evidence (screenshots under `plans/ux-verified-technical-fixes/agent-runs/shots/`):

- `s16-b-375-1-top.png`, `s16-b-375-2-title-focus.png`, `s16-b-375-3-body-focus.png`, `s16-b-375-4-bottom.png`
- `s16-b-768-1-top.png`, `s16-b-768-2-title-focus.png`, `s16-b-768-3-body-focus.png`, `s16-b-768-4-bottom.png`

Measured button geometry (the only "Save draft" control on the page):

| Viewport | State              | Save top (px) | Viewport h | In viewport?    |
| -------- | ------------------ | ------------- | ---------- | --------------- |
| 375x812  | page top           | 1519          | 812        | no (below fold) |
| 375x812  | title focused      | 1519          | 812        | no (below fold) |
| 375x812  | body focused       | 1519          | 812        | no (below fold) |
| 375x812  | scrolled to bottom | 277           | 812        | yes             |
| 768x1024 | page top           | 1435          | 1024       | no (below fold) |
| 768x1024 | title focused      | 1435          | 1024       | no (below fold) |
| 768x1024 | body focused       | 1435          | 1024       | no (below fold) |
| 768x1024 | scrolled to bottom | 489           | 1024       | yes             |

The button uses `position: static` (not sticky/fixed), `display: flex`,
`visibility: visible`. Page is scrollable (scrollHeight ~2054 / ~1970).

Root cause: `NewsEditorForm` lays out as
`grid gap-8 lg:grid-cols-[minmax(0,1fr)_320px]`
(`src/components/admin/NewsEditorForm.tsx:115`). The Save/Publish controls live
in the right-hand `<aside>` "Publish" card (`NewsEditorForm.tsx:214-235`). Below
the `lg` breakpoint (1024px) the grid collapses to one column and the aside
stacks **after** the entire title + slug + excerpt + body editor. So while a
user is actually typing in the title or body, "Save draft" is ~1500px down the
page and never in view; they must scroll past the whole form to find it.

Fix sketch: give small/medium screens a persistent Save affordance. Either
(a) a `sticky bottom-0` mobile action bar containing Save draft / Publish that
shows below `lg` (mirrors the existing `MobileEditorActionBar` pattern used by
the SEO editor), or (b) make the Publish aside `lg:sticky lg:top-...` AND add a
compact sticky save bar for `< lg`. Touches `NewsEditorForm.tsx` only; verify by
re-running this script and confirming `inViewport: true` during title/body
focus at 375 and 768.

## Check C — SEO-P3d create-gate click reliability on `/admin/pages/new`

**Verdict: NOT-REPRODUCED** under the specified protocol (click after the load event). 20/20 (and an initial 10/10) clicks registered cleanly; gate dismissed every time; zero misses, zero errors.

Evidence:

- realistic loop (`s16-check-c.mjs`, click after `load` with normal Playwright
  click): **20/20 SUCCESS**, 0 miss, 0 error.
- No draft page is created by the gate click — `onChoosePageTemplate`
  (`useSeoPageEditorController.ts:1201-1212`) only sets client state
  (`setHasSelectedNewPageMode(true)`, replaces content); there is no DB insert
  or network call at gate time. So no cleanup is required and none was performed.

Nuance worth recording (not a reproduction, but informs the fix decision):

- A **strict** variant (`s16-check-c-strict.mjs`) that bypasses Playwright's
  actionability auto-wait and fires a raw `element.click()` at `domcontentloaded`
  dropped **6/10** clicks. The "Start building page" button is a client-only
  `<button onClick={handleStartBuilding}>` with no progressive-enhancement / SSR
  fallback (`SeoPageEditorShell.tsx:152-159`), so a click landing in the window
  before React hydration attaches the handler is silently lost.
- This race is **not reachable by a human** at the timing the review described
  (a person cannot click in the sub-hydration window after the page paints), and
  the as-specified protocol is clean. I'm marking it NOT-REPRODUCED for the
  scheduled-fix decision, but flagging the latent client-only gate so the
  orchestrator can decide whether a defensive hardening is worthwhile.

Fix sketch (only if the orchestrator wants the latent race closed, low priority):
disable the button until hydration/mount (`useEffect`-set `ready` flag) or keep
the click handler but render the gate's primary action inside a `<form>` so a
pre-hydration activation still does something. Given the realistic verdict is
clean, no fix is required for S16's purposes.
