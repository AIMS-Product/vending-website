# S8 — Redirect path format validation

Node: S8 (Tier T2) — finding P2-3 from the SEO-builder review.
Branch: ux-review-fixes.

## Goal

`sourcePath`/`destinationPath` were validated only with `.min(1)`, so a typo
like `foo` passed the form layer and hit the service's jargon error
("Path must be root-relative."). Add a form-layer format floor that mirrors the
service's ACTUAL accepted format and surfaces the friendly inline field error
before the service is reached — without false-rejecting anything the service
genuinely accepts.

## Exact service rules mirrored (read-only, src/lib/services/seo-pages.ts)

Source path (`normalizeSourcePath`, ~L1634):

- `normalizeInternalPath` → `isRootRelativePath` (L1690): must `startsWith("/")`,
  NOT `startsWith("//")`, contain no `\`, no `%5c`.
- THEN `isBuilderRoutePath` (page-paths.ts L99): must be `/{resources|blog|landing|videos|solutions}/{slug}`.

Destination path (`normalizeDestinationPath`, ~L1662):

- Accepts a root-relative path (`isRootRelativePath`), OR
- a valid `http:`/`https:` URL (`new URL(...)` with http(s) protocol).
- Otherwise → "Redirect destination must be an internal path or http(s) URL."

### What I mirrored at the form layer, and why only this much

I deliberately mirrored only the **leading-slash floor** (`startsWith("/")`),
plus the http(s)-URL allowance for the destination. This is the narrowest check
that:

- catches the common `foo` typo the finding is about, and
- never false-rejects a value the service accepts (every service-accepted
  source/destination either starts with `/` or — destination only — is an
  http(s) URL).

I did NOT replicate the stricter builder-route-shape, backslash, `%5c`, or
self-redirect checks at the form layer. Those stay in the service so its
specific messages still surface (e.g. `/about` → "Builder redirect sources must
be builder page paths."; `/blog\evil` → destination error). Duplicating them
in the form would risk divergence and false rejects with no user benefit.

## Change (src/app/admin/pages/redirects/actions.ts only)

- Added `SOURCE_PATH_HINT` / `DESTINATION_PATH_HINT` message constants.
- Added `isRootRelativeLike(value)` (`startsWith("/")`) and `isHttpUrl(value)`
  (http/https via `new URL`) helpers, each with a comment tying it to the
  service rule it mirrors.
- `createSchema.sourcePath`: added `.refine(isRootRelativeLike, SOURCE_PATH_HINT)`.
- `createSchema.destinationPath`: added
  `.refine(v => isRootRelativeLike(v) || isHttpUrl(v), DESTINATION_PATH_HINT)`.
- `updateSchema` extends `createSchema`, so update inherits the same floor.

No service files touched. No form-component change needed: `RedirectCreateForm`
and `RedirectRow` (edit form) already render `<FieldError>` inline per field and
preserve echoed `values` via `defaultValue` + a `key={JSON.stringify(values)}`
remount. The existing `zodErrorState` already maps `sourcePath`/`destinationPath`
issue keys onto the right field.

### Existing test updated (not weakened)

`maps a service validation issue onto the destination field` previously used
`destinationPath: "not-a-path"` and relied on it reaching the service. That value
is now correctly caught at the form floor (intended behavior change). To keep the
test's actual intent — service issue → destination field mapping — I changed the
input to `/blog\evil`, which passes the leading-slash floor but the service
rejects. This also fixed transient test pollution: the old version left an
unconsumed `mockRejectedValueOnce` (service no longer called) that bled into the
next service-reaching test.
