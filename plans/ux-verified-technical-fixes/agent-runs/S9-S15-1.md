## NODE S9 — Create-page success feedback (C140)

### Investigation

Create flow: `/admin/pages/new` → editor → first manual save runs `saveSeoPage`
(actions.ts), which on the create branch `revalidatePath("/admin/pages")` then
`redirect("/admin/pages/<id>?saved=1")` — i.e. it intentionally lands the admin
back **in the editor** (the `?saved=1` toast + the S3b autosave/`history.replaceState`
architecture all depend on staying in the editor). The admin returns to the list
manually via the editor top-rail "Pages" link, which is a plain `/admin/pages`
with no signal. That is the C140 race: on the list there was no confirmation the
page was created and no quick way to find the new row.

Default list sort is `updated-desc` (seo-pages-list.ts), and a just-created page
has the freshest `updated_at`, so it already sorts to the top — no sort change
needed (acceptance criterion satisfied by proving, not changing, sort).

### Smallest honest pattern chosen

A deterministic, param-driven banner keyed on `?created=<id>` (mirrors the
existing bulk-archive `?archived=N` banner pattern and the redirects `?created=1`
pattern), plus a highlight on the matching row. Param is validated against the
real page list (`resolveCreatedPage`) so a stale/spoofed id renders nothing.

- `CreatedPageBanner` — `role="status"`, names the page, "Open page" link to
  `/admin/pages/<id>`.
- `data-created-row="true"` + emerald highlight on the matching desktop row and
  mobile card.
- Top-of-list guaranteed by existing `updated-desc` default (proven by test).

### RGR

- RED: 3 new tests failed (banner+Open link, highlighted row, KPI>0); the two
  negative-assertion tests passed pre-implementation (correct — nothing rendered).
- GREEN: implemented banner + row highlight; all tests pass.
- REFACTOR: extracted `resolveCreatedPage`; reused emerald tone already used by
  the bulk-archive success banner for visual consistency.

### Acceptance criteria

- [x] Deterministic success feedback (param-driven, not a race).
- [x] Feedback includes "Open page" link to `/admin/pages/<id>`.
- [x] New row findable at top of default order — proven by `updated-desc` test
      (no sort change), plus the row highlight.

### CROSS-BOUNDARY GAP (integrator action required)

The list-side mechanism is complete and proven, but the real create flow does
NOT yet emit `?created=<id>` onto the list URL. The emission point is the editor
top-rail "Pages" back-link in
`src/components/admin/seo-page-editor/SeoPageEditorTopRail.tsx`
(line ~95: `<Link href="/admin/pages">`), which is OUTSIDE this worker's write
boundary. Verified on :3001 that this link is currently `href="/admin/pages"`
with no param, so the banner does not fire after a real create until that link
carries `?created=<effectivePageId>` when the page was just created in-session.
Recommended one-line change for the owner of that file: make the back-link href
`/admin/pages?created=${effectivePageId}` when `createdDraftId` is set (or the
session created the row). Until then S9's banner only activates on a URL that
already carries the param.

from `countPagesByStatus`, which does not count schedule-failed. Reused the same
source-of-truth: counted `scheduled_publish_status === "failed"` over the same

## NODE S20 — Top-rail back-link carries ?created (closes the S9 e2e gap)

Extended boundary granted: `SeoPageEditorTopRail.tsx` (+ its test). Controller
NOT modified — the needed state was already exposed.

### What changed

- Added pure helper `backToPagesHref(page, effectivePageId)` exported from
  `SeoPageEditorTopRail.tsx`:
  - just-created in-session (`!page?.id && effectivePageId`) →
    `"/admin/pages?created=<effectivePageId>"`
  - loaded existing page, or new page with no row yet → `"/admin/pages"`
- Wired the top-rail "Pages" back-link to use it
  (`href={backToPagesHref(page, effectivePageId)}`). `effectivePageId` is
  already returned by the controller (line ~873), so no controller change was
  needed — the boundary stayed read-only there.

### Why this signal

"Just created in-session" = the editor opened on `/admin/pages/new` (no loaded
`page`) AND a draft row now exists (`effectivePageId`, set by S3b auto-create or
the first save). Editing a loaded page has `page.id` set → plain link, so
returning to the list from an edit never shows a stale created banner.
