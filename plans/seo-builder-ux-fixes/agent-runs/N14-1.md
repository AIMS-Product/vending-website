## Root Cause / Investigation (wiring decisions â€” no forked logic)

The acceptance bar is that Cmd+Enter behaves EXACTLY like clicking Publish (blocker-gated + confirm-gated). I traced the real affordances:

- Save: `SeoPageEditorForm` / top rail / my N11 mobile bar all submit via `form.requestSubmit(<button[type=submit][name=intent][value=save]>)`. The hook reuses that exact path. `preventDefault()` on the keydown suppresses the browser's native Save dialog.
- Publish: `PublishButton` (SeoPublishPanel) owns the gate in its own `onClick` â€” if `publishDisabled` it reveals blockers and returns; else it opens the confirm dialog. The actual publish submit only happens via the "Confirm publish" button inside the confirm dialog. So the hook simply CLICKS the real Publish button (found by matching `editor.publishButtonLabel` text within the form). This runs the identical gate; zero publish logic is duplicated. If the SEO panel is collapsed (button not mounted), the hook calls `toggleSeoSidebar()` (the route pre-approved for N11) and clicks on the next animation frame.
- Block picker: `BlockPicker` manages its own open state and is opened only by its own trigger (`[data-testid="block-picker-trigger"]`); the controller exposes no open function. So "/" clicks that real trigger. Typing-suppression: `isTextEntryElement()` returns true for input/textarea/select/contenteditable, so "/" passes through to content there.

The controller does NOT expose `formRef`; rather than edit n1's controller, the hook locates the form via the existing Save button's `.form` property (a stable, single-purpose affordance).
