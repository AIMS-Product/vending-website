## Root Cause / Investigation

The editor's sticky top rail (`sticky top-0 z-50`) does NOT stick to the viewport on mobile. A DOM diagnostic showed the rail's nearest scrolling ancestor is the Workspace root `div` (`relative … overflow-x-hidden …`). Per CSS spec, `overflow-x: hidden` promotes `overflow-y` from `visible` to `auto`, turning that div (height 2040px) into a scroll container. `position: sticky` sticks to that container, not the viewport — so as the body scrolls, the rail scrolls out of view with it. The same trap exists on an ancestor admin shell (`min-h-screen overflow-x-hidden`) that is outside my write scope, so I could not fix this by removing overflow.

Fix decision: a `position: fixed` bar. I verified no `transform`/`filter`/`contain`/`perspective` ancestor exists between the bar and the root, so `fixed` correctly anchors to the viewport. Per the page-builder design contract, Save draft lives in the top rail and Publish lives in the right SEO panel — so the mobile bar mirrors the Save submit (same `name="intent" value="save"`, identical submit path — not a forked action) and routes to Publish by opening the SEO panel via the existing `toggleSeoSidebar`, rather than duplicating the publish submit.

The SEO panel was already collapsible in round-1 (the top-rail "SEO" toggle drives `mobileEditorPanel`/`isSeoSidebarCollapsed`); N11 keeps that and adds a second, always-on-screen entry point to it.
