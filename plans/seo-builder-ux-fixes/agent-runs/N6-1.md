## Root Cause / Investigation

- **"Explicitly saved" signal (the DONE-vs-BLOCKED crux):** the schema has no explicit-save column, and an explicitly-saved draft is indistinguishable from an S3b auto-draft at the row level (both `status='draft'`, `published_at=null`). The distinguishing signal is derivable **without editing the controller**: the controller already exposes `page` (SSR prop), `effectivePageId` (`page?.id ?? createdDraftId`), `state` (the `saveSeoPage` action result), and `savedFromRedirect`. Auto-created-never-saved = `!loadedPageId && effectivePageId && !explicitlySaved`, where `explicitlySaved = state.status === "saved" || savedFromRedirect`.
- **Mid-run correction:** my first browser run exposed that an explicit Save on an already-auto-created row does NOT redirect/remount (no `?saved=1`, `page` stays undefined). Without the `state.status === "saved"` term the guard would keep offering Discard on a row the user just saved — an invariant violation. Folding in `editor.state.status` (already returned by the controller) fixed it; re-verified green.
- **In-app nav interception:** Next 16.2.6's `<Link onNavigate>` is the canonical block API, but the editor's "Pages" link lives in `SeoPageEditorTopRail.tsx` (out of scope). A capture-phase document `click` listener intercepts the underlying `<a>` before next/link's handler runs (`preventDefault` + `stopPropagation`), only for plain same-origin internal navigations. Read `node_modules/next/dist/docs/01-app/03-api-reference/02-components/link.md` to confirm the current API before implementing.
