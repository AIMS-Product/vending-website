## Root Cause / Investigation

Axe `aria-prohibited-attr` (SERIOUS) fired because `aria-label` is prohibited on an element whose (implicit) role is `generic` — a bare `<span>` with no `role`. The persona scan reported `span[aria-label="Hero block"]` / `"FAQ block"` etc. I traced these to `BlockToolbar` in `BuilderEditorUi.tsx`, the per-block canvas chrome header:

- L276–281: block-type icon wrapper `<span aria-label={`${typeLabel} block`}>` (→ "Hero block", "FAQ block", …).
- L297–302: non-Ready status dot `<span aria-label={status}>`.
  Both are decorative-glyph containers (`BuilderGlyph` SVG / a colored dot) whose only accessible name is the `aria-label`. Adding `role="img"` makes naming permitted while keeping the exact same announced name.

I audited every other dynamic `aria-label` in `src/components/admin/seo-page-editor/` (grep). All remaining ones sit on `<button>` / `<input>` / `<a>` elements where naming is permitted (e.g. `SeoPageEditorShell.tsx:625` button, `AiBuilderAssistant.tsx:1917` checkbox, `BuilderEditorUi.tsx:107` button). The two BlockToolbar spans were the only offenders, consistent with the axe finding.
