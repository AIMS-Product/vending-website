## Root Cause / Investigation

Original disease: blockers surfaced one-at-a-time / via a hover-only `title`
tooltip, the chip used `seoReadiness.blockers.length`, the disabled button used a
separate `title` string, and the readiness panel showed a `slice(0,6)` of mixed
findings — three independent representations that could disagree and no
single upfront list. Fix: one derivation (`derivePublishBlockerChecklist`)
mapping `summary.blockers` 1:1 to human-readable items with a target ref;
`publishDisabled` and the chip now read its length.

Investigation note (parity sanity check): a raw `createPageBlock("cta")` yields
both `missing_cta_label` and `missing_cta_tracking`, but a CTA inserted through
the real editor runs `withEditorDefaultsForNewBlock`, which defaults
`href:"/apply"` and `trackingName:"cta"`, so the live editor only blocks on the
empty label. The browser checklist matching that exactly (6 items, not 7)
confirms the list is a faithful 1:1 of `summary.blockers` — not a dropped item.

## Behavior Preservation

Rule parity (no readiness rule added or removed):

- The checklist is `summary.blockers.map(...)` plus the synthesized "save the
  draft first" item, which is shown only when `!canPublish` — a pre-existing
  publish precondition (the old `publishDisabled = !canPublish || blockers > 0`
  already blocked on it), not a new rule.
- `publishDisabled` is now `publishBlockerChecklist.length > 0`. This is
  boolean-identical to the old `!canPublish || seoReadiness.blockers.length > 0`:
  the list has `blockers.length` items plus one iff `!canPublish`, so its length
  is `> 0` exactly when the old expression was true. Same set of conditions
  blocks publish before and after.
- Unit test `preserves rule parity` asserts the rendered checklist's rule codes
  (excluding the save-first step) equal `summary.blockers` codes exactly.

Confirm-before-publish flow: untouched — `PublishConfirmDialog` and the
`onRequestConfirm` path are unchanged; the disabled button still calls
`revealPublishBlocker` (now focusing the first checklist item) and never
submits. Debounced autosave and the scheduler were not touched.

## Residual Risk

- The Action Items panel in `SeoReadinessPanel` now lists only warnings +
  opportunities. If a downstream consumer expected blockers in that `slice(0,6)`
  list, it would no longer see them there — but blockers are now always visible
  in the dedicated checklist above, which is the intended single home. No test
  or caller depended on blockers being in Action Items.
- The save-first deep-link targets the top-rail Save button via
  `button[name="intent"][value="save"]` (an existing element, not edited). If
  that selector ever changes in the top rail it would no-op the focus; low risk.
