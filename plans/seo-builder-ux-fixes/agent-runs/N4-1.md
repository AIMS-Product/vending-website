## Root Cause / Investigation

Two defects:

1. No retry on autosave failure. The debounced autosave effect fired once after
   1200ms; on failure it set `{status:"error", message:"Autosave failed."}` and
   STOPPED — nothing retried until the user made another edit. The message was
   also vague ("Autosave failed.") and gave no manual-save guidance, so a user
   could believe nothing was wrong or not know their work was unsaved (the
   data-loss lie this review targets).
2. Duplicate "saved" announcements. After a manual save the action-state toast
   ("Draft saved.") co-occurred with the separate autosave "Autosaved …"
   indicator — two "saved" surfaces for one save.

Fix: (1) a capped backoff retry loop in the autosave effect driven by a pure
policy module — retry at 2s/5s/10s, then rest until the next edit or manual save
(no request storm); the error message now always says the work is NOT saved and
offers Save draft. (2) On a manual save/publish submit, clear the autosave state
and reset the retry budget, so the autosave "saved" indicator no longer
co-announces with the manual toast; the toast node is keyed by message so it
announces once.

## Behavior Preservation

- Debounced autosave semantics unchanged: still a 1200ms debounce on the first
  attempt per edit; requests are still serialized through `autosaveInFlight` so
  overlapping saves can't commit out of order. The retry reuses that same chain.
- Manual save path unchanged (same submit → `saveSeoPage`); I only additionally
  clear the transient autosave indicator on manual submit.
- No request storm: retries are capped at 3 (2s/5s/10s) then rest until the next
  edit or manual save re-arms the budget — proven by the policy unit tests
  (`nextAutosaveRetryDelayMs` returns null at the cap) and by the recovery
  behaviour in the browser.
- Data-loss lie fixed: the failure message never says "saved" (unit-asserted)
  and was browser-verified as "Couldn't save — retrying. Save manually…".
- Confidence: 93% (effect-level retry/cap proven via policy unit tests +
  browser recovery; the precise cap-then-rest timing is exercised by the policy
  tests rather than a hook harness).
