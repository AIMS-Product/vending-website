## Root Cause / Investigation (how auto-start was triggered)

Auto-start lives in the **controller** (`useSeoPageEditorController.ts`, n1-active — forbidden): `onChoosePageTemplate` (create flow) calls `shouldOfferBuilderWalkthrough()` (true when no `page-builder-editor-walkthrough-seen` localStorage flag) and, if so, `applyWalkthroughPanelLayout(1)` + `setBuilderWalkthroughStep(1)`. `BuilderEditorWalkthrough` then rendered the overlay whenever `editor.builderWalkthroughStep` was truthy. So a first-time user got the tour popped on top of the editor automatically.

The controller exposes `builderWalkthroughStep`, `advanceBuilderWalkthrough`, `finish/dismissBuilderWalkthrough` — but **no "start from null"** function, and the auto-start call sits inside the forbidden controller.

## Design (no controller edit, no NEEDS ATTENTION)

Rather than gate the controller-driven tour (which would need a controller change to start it on demand), `BuilderEditorWalkthrough` now owns the whole tour locally:

- Local `activeStep` state (null → no tour). The component **ignores `editor.builderWalkthroughStep`**, so the controller's auto-start can set it to 1 all it wants — nothing renders. This neutralizes the intrusive auto-start entirely from within my file.
- A persistent floating "Quick tour" launch button (the opt-in affordance, `aria-label="Start the quick tour"`) renders whenever the tour isn't active.
- Click → `activeStep = 1`, expanding the relevant panel via the controller's exposed `toggleBlockSidebar`/`toggleSeoSidebar` (read `isBlockSidebarCollapsed`/`isSeoSidebarCollapsed`) so the highlight anchors to a real target.
- Next advances 1→2→3 (revealing each step's panel); "Got it"/Skip end the tour, persisting the seen-flag (same key the controller checks, so the legacy auto-start also stays off) and returning to the launcher so it is re-runnable.
- Positioning is unchanged — the prior fix is preserved.

A self-contained component-only solution; no `new/page.tsx`/Shell change, no entry-wiring gap, no NEEDS ATTENTION.
