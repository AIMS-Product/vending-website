The brief's part-(a) hypothesis ("saved value not hydrated back into the form;
field re-renders empty") does NOT reproduce on current committed code. Full
chain traced and verified end-to-end:

- Write: `saveSeoPage`/`autosaveSeoPageDraft` → `scheduledPublishMetadataFromPageForm`
  → `adminSaveSeoPageDraft` → `buildSeoPageDraftPatch` maps `scheduledPublishAt`
  → the `scheduled_publish_at` column. Correct.
- Read: `adminGetSeoPageById` selects `scheduled_publish_at` (in
  `SEO_PAGE_FIELDS`); `/admin/pages/[id]/page.tsx` passes the full `page` →
  controller → `GovernanceFields`/`ScheduleField` baseline =
  `formatDateTimeLocalInTimeZone(page.scheduled_publish_at)` used as the field's
  `defaultValue`. Correct.
- Browser repro (throwaway drafts, cleaned up): new page → set
  `2026-06-12T09:00` PT → save → reload → field shows `2026-06-12T09:00` (DB
  stored `16:00Z`, correct PDT conversion). Editing an existing scheduled page:
  field hydrates from the DB, and survives autosave in both the field and the
  DB columns (autosave leaves scheduler columns untouched because the submitted
  value equals the mount-time baseline).

Conclusion: the round-trip was already fixed by the June 10 scheduler-ownership
rework (migration `20260610091000`), which introduced the
`scheduledPublishAtBaseline` hidden field and the lazy `useState` baseline. The

stale. Lead confirmed this reading and approved locking (a) and shipping (b).
