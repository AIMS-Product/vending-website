## Root Cause / Investigation

Before N3, on a successful publish the controller called `refresh()` (re-fetch
the server component) but the confirm dialog's `isConfirmingPublish` was LOCAL
to `PublishStatusSection` and was only reset on Cancel — `refresh()` re-renders
without remounting client components, so the confirm card could stay armed
after a publish (the "accidental re-publish" risk). There was also no success
surface and no live-link, so a user had no in-panel confirmation or a safe way
to open the new live page (and a naive link would 404 during the post-publish
propagation window).

Fix: (1) lift `isConfirmingPublish` to the controller and dismiss it inside
`handleEditorFormSubmit` when the publish is submitted — so the card is gone the
moment publish fires and any re-publish must re-open a fresh confirm; (2) a
`PublishSuccessCard` that polls the route and only reveals "Open live page" once
it responds 200, with a "may take a moment" fallback otherwise.

## Residual Risk

- The success card stays visible while `publishJustSucceeded` is true (until the
  user edits — which resets the action state — or navigates). This is intended
  (persistent success confirmation), and the transient toast handles the
  immediate feedback. Low risk.
- The poller uses HEAD with `cache: "no-store"`; if a route only supports GET it
  would read not-live and fall to the "may take a moment" copy rather than a
  false link — a safe failure mode. The browser gate confirmed HEAD returns 200
  for the published resource route.
- `SeoPublishPanel.tsx` is 864 lines (pre-existing largeness); I extracted the
  success card rather than inlining. Splitting the panel is out of N3 scope.
