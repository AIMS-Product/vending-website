## Root Cause / Investigation (AM/PM mismatch) — stated before fix

There were **three** divergent date formatters:

- Revision **list** (`SeoPageRevisionPanel.formatDateTime`): `toLocaleString("en-US", {…, timeZone: "UTC"})` — **UTC**, 12-hour.
- Revision **detail** (`[revisionId]/page.tsx.formatDateTime`): same options but **no `timeZone`** → runtime-local (UTC on Vercel), 12-hour.
- **Editor** (`SeoPageEditorTopRail.formatRailTime`): `toLocaleTimeString` with **no `timeZone`** → runtime-local, 12-hour.

**Root cause: a timezone divergence, not a 12/24h one.** All three are 12-hour, but the list forced UTC while the others used an unspecified (local) zone. For this Pacific-Time business (`SCHEDULED_PUBLISH_TIME_ZONE = "America/Los_Angeles"`, already used by the scheduled-publish UI), the same instant renders hours apart and flips AM/PM between the list and the editor/detail page.

**Fix (one shared formatter, per the brief's preference):** `formatRevisionDateTime` renders the business's Pacific Time deterministically with a `PDT/PST` zone label, matching the scheduled-publish UI. Both revision surfaces (list + detail) now use it, so they agree with each other and with the scheduled-publish display. The editor's `formatRailTime` lives in `SeoPageEditorTopRail.tsx` (n8-active) — I did NOT edit it; unifying it onto this formatter is flagged as a handoff below.
