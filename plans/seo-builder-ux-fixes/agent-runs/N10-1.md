## Uniqueness scope (investigated)

The slug `unique` constraint has evolved: the original global `slug unique` was
dropped (20260601110000), replaced by a partial slug index, then dropped again
and replaced by `seo_pages_active_route_path_unique_idx` on `route_path`
`WHERE status <> 'archived'` (20260602103000). So the real uniqueness scope is
**route_path among non-archived pages** — archived pages do NOT reserve a slug.
My collision check (`isActiveRoutePathTaken`) queries exactly that scope:
`select id from seo_pages where route_path = $1 and status <> 'archived'`.
`route_path = pagePathForSlug(slug, routePrefix)`.

slugSchema caps slugs at 120 chars (pattern `^[a-z0-9]+(?:-[a-z0-9]+)*$`); the
helper truncates the base before the suffix to stay within 120.

## Edge cases covered

- Duplicate of a duplicate: `copyBaseSlug` strips the trailing copy marker, so
  `start-vending-copy` → base `start-vending` → next free `…-copy-2`. Browser-
  proven (`-copy-2`, never `-copy-copy`).
- Very long source slugs: base is truncated to `120 - suffix.length` and any
  trailing hyphen trimmed so the result is ≤120 and valid.
- Collision-loop bound: 50 sequential attempts, then a `…-copy-{timestamp}`
  fallback (re-checked once); effectively collision-free.

## Incident during browser gate (recovered)

To get a clean env (the shared :3001 editor route was throwing a JSX parse
error from N16's mid-edit SeoPageRevisionPanel.tsx:264), I briefly started a
SECOND `next dev` on :3002 against the same `.next` dir. Two dev servers sharing
one `.next` corrupted the turbopack SST cache, 500'd `/admin/pages`, and crashed
both servers. Recovered by killing :3002, removing
`.next/dev/cache/turbopack` + the stale lock (regenerable cache only — no
source touched), and restarting the single shared :3001 (verified root 200 +
`/admin/pages` 200). Reported to the lead. Lesson: never run a 2nd dev server
against the shared `.next`.

The first "duplicate failed (?error=duplicate)" runs were NOT a code bug — my
DB-seeded source fixture omitted `page_type`/`template_key`, so
`parseTemplateSelection` (inside `adminCreateSeoPage`) rejected the null page
type. Real pages always have these; fixing the fixture made the duplicate
succeed. The slug-derivation change itself was correct throughout.
