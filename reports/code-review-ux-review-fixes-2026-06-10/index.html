Findings (ranked)

MAJOR "Save draft" silently does nothing when the title is empty and the Readiness tab is active regression · I14

Where

SeoPanelTabs.tsx:81-93 · SeoPublishPanel.tsx:342-385 · SeoPageEditorForm.tsx:66-71

What happens

The new tabs keep the Settings panel mounted but hidden. It contains <input name="title" required> and <input name="slug" required>. The browser still validates hidden controls — but when an invalid control can't be focused, Chrome aborts the form submission with only a console warning. No toast, no bubble, nothing. Affects the top-rail Save, the mobile bar, Cmd+S, and the unsaved-exit dialog's "Save draft".

Explainer: before this round the settings fields were always visible, so an empty title produced a browser validation bubble pointing at the field. Now the field is invisible, the browser refuses to submit a form with an invisible invalid field, and it tells no one. Weirdly, if you collapse the SEO panel entirely the save works (only validation-exempt type="hidden" mirrors remain) and the server replies "Title needs at least 3 characters." — so the feedback depends on which panel state you happen to be in.

Fix direction

Drop required from those two inputs — the server already validates and the toast surfaces the message (matches the collapsed-panel path). Alternatives: noValidate on the form, or an invalid-event handler that activates the Settings tab.

Verify

New page → leave title empty → Readiness tab active → click Save draft → must produce visible feedback.

MAJOR Publish-blocker checklist click-to-fix is a no-op for every SEO-settings blocker regression · I14 vs I1/I19

Where

useSeoPageEditorController.ts:1175-1188 (focusPublishBlocker) · :1207-1220 (focusSeoSetting) · SeoPanelTabs.tsx

What happens

Checklist items like "Add a page title" target element ids (page-title-field, seo-title-field, …) that live inside the hidden Settings tab. scrollIntoView() and focus() on a display:none element do nothing — and SeoPanelTabs owns its active tab internally with no API to switch it. Same dead end for the verdict card's "fix next" and the readiness findings' "Open settings".

Explainer: the canonical checklist (the flagship I1/I19 fix) is correct — count, list, and disabled button can't disagree. But its whole point is "click the item, land in the field that fixes it", and that navigation is dead in the panel's default state (Readiness tab first). Canvas/block targets and the "save first" item still work; it's specifically the SEO-settings class — the most common blockers — that go nowhere.

Fix direction

Make SeoPanelTabs controllable (activeId / onActiveIdChange) and have both focus functions activate the Settings tab before the rAF focus.

Verify

Empty SEO title, Readiness tab active → click "Add an SEO title" in the checklist → focus must land in the SEO title field.

MODERATE Saving with the SEO panel collapsed wipes governance & social metadata pre-existing · exposure ↑ this round

Where

SeoPageEditorWorkspace.tsx:62-64 · actions.ts:816-824, 943-954 · SeoPublishPanel.tsx:588-718

What happens

Collapsing the panel unmounts it. The governance fields (internal tags, topic cluster, campaign, funnel stage, lifecycle, review period/date, OG title/description) are uncontrolled and exist only there — no hidden mirrors. parsePageFormData defaults the missing values and the save writes them: tags → [], lifecycle → drafting, review period → 6.

Explainer: autosave was protected on purpose (it falls back to the loaded row's values — proof the authors knew unmounted fields vanish) and the scheduler is protected by its baseline trick. The manual-save path just never got the same treatment. The unmount predates this round, but N11's always-visible mobile Save button (panel collapsed is the default on phones) and the new Cmd+S make "save while the panel is unmounted" a first-class path now.

Fix direction

Mirror governance into controlled state + hidden inputs like the SEO fields, or make the server treat absent governance keys as "no change".

Verify

Set internal tags → save → collapse panel → save again → tags must survive.

MODERATE Never-saved-delete floor depends on a best-effort revision snapshot Tier 1 · narrow

Where

actions.ts:852-864 · seo-page-drafts.ts:42-55 · unsaved-exit-guard-state.ts:73-82

What happens

The server decides "never explicitly saved" mainly via revisionCount === 0 — and the manual-save revision snapshot that creates that count deliberately swallows failures ("must never fail the user's save"). If that insert fails once, an explicitly-saved draft looks never-saved and Discard would hard-delete it.

Explainer: real mitigations exist — the client disarms Discard after a successful save, and the check-then-delete race is closed at the DB (the page_revisions.page_id FK has no CASCADE, so a page that gains a revision mid-race fails the delete). Exploiting this needs a snapshot failure plus a stale client or a misused action. Still, a Tier-1 delete floor shouldn't rest on a write that's allowed to fail.

Fix direction

Persist an explicit first_saved_at-style fact transactionally with the save, or make the snapshot non-optional for the first manual save.

Verify

Unit test: mock the revision insert to fail during saveSeoPage(intent: "save"), then assert deleteNeverSavedSeoPageDraft returns protected.

MODERATE Typing in the schedule field arms the schedule via autosave — no save needed, and the new status card won't show it pre-existing · undermines I2

Where

useSeoPageEditorController.ts:135-147 (autosave metadata field list) · actions.ts:973-999

What happens

scheduledPublishAt is in the autosave metadata listener, so editing it triggers a debounced autosave whose value differs from the mount baseline → the server writes scheduledPublishStatus: "scheduled". Clearing the field symmetrically cancels. The UI copy says "Save the draft to arm the schedule", and the new ScheduleStatusCard reads server props that don't refresh on autosave — so the honesty surface can be silently stale against the DB while a publish is armed.

Explainer: worst case: an admin types a tentative time while thinking, navigates away without saving, believes nothing is scheduled — and the page publishes itself at that time. The root cause predates this round (field list and server path are unchanged), but it directly contradicts the I2 "schedule status honesty" deliverable this round shipped.

Fix direction

Remove scheduledPublishAt / cancelScheduledPublish from autosaveMetadataFieldNames so only explicit saves carry schedule changes.

Verify

Type a future time, don't save, wait 2s → DB row must still show no schedule.

MODERATE Bulk archive: silent partial failure, no cap, no dedup N19/I20

Where

actions.ts:765-793

What happens

If any id succeeds you're redirected back to the list with no indication which failed (failures live only in server logs); invalid ids are silently dropped; the id list is unbounded and un-deduplicated (sequential RPCs — a timeout leaves a half-archived batch with no visible record).

Fix direction

new Set(ids), cap (~50), and carry ?archived=N&failed=M in the redirect.

MINOR Redirect "internal path" classification bypassable with a backslash

Where

seo-pages.ts:1686-1701 · served via redirects.ts:5 / proxy.ts:103-106

/\evil.com passes the leading-/ check and the DB CHECK, but new URL("/\\evil.com", base) resolves to https://evil.com/ — a 301 off-site from a path stored and displayed as internal. MINOR Unsaved-exit guard misses SPA back/forward; "Save & leave" doesn't await the save

Where

useUnsavedExitGuard.ts:112-162

The guard intercepts anchor clicks and beforeunload only — browser Back is a popstate SPA navigation and bypasses it (impact bounded to an orphan draft; autosave protects content). Cheap fix: add a popstate listener while armed; gate the push on the save starting/succeeding.
